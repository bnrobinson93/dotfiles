Apply this skill only for prose-heavy work where style matters.

Do not use for:
- routine implementation notes
- terse chat replies
- mechanical summaries
- ordinary code edits where prose quality is not central

Default process:
1. Identify audience and artifact: README, design doc, comment, essay, note, post.
2. Draft with clarity first, style second.
3. Tighten aggressively. Cut filler, hedging, throat-clearing, and repeated points.
4. Read for rhythm. If it sounds like writing, rewrite it.
5. End cleanly. No summary paragraph unless user explicitly wants one.

Core voice:
- Co-pilgrim, not lecturer.
- Concrete beats abstract.
- Conversational, but not sloppy.
- Short sentences win.
- One main point per piece.
- Precise nouns and verbs over adverbs and puffery.

Editing rules:
- Cut filler words such as `very`, `really`, `just`, `basically`, `simply`, `actually`.
- Prefer plain words over inflated ones.
- Use active voice unless passive is clearly better.
- Put the important word near the end of the sentence.
- Keep intros short. Get to the point fast.
- Do not over-explain what the reader already knows.
- Do not close with `In conclusion` or a recap unless requested.

When nuance, samples, or the fuller voice matters, read:
- `~/.codex/WritingVoice.md`

Use that file as the canonical detailed guide. Keep this skill lean.
